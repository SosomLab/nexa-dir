# refactor/002 통합 감사 리포트 (전체 개발 범위 종합 점검)

> 1차([20260702_130615_refactor-001-audit.md](20260702_130615_refactor-001-audit.md)) 방식 차용 — 병렬 감사 5축(성능·설계수용성·구조리팩토링·문서정합·Double Commander 비교) 결과를 통합.
> 분기 기준: `b38e6b3`(main, C1 병합 직후 · 태그 `0.1.0`=병합 전 베이스라인).
> **원칙**: ① 1차에서 **완료/취소/정리된 항목은 제외**하고 미해결·이월·신규만으로 2차 기준 수립. ② C# UI 스택을 **Double Commander(네이티브)** 와 수치 비교.

## 0. 총평

**설계는 훼손되지 않았고, C1(코어 트리 이관)으로 오히려 개선됐다.** 1차의 최대 문제(A1 플래그십 O(n)·NFR 위배, A3 ABI 미검사, C6 죽은 코드)는 실제로 해소됐고, 진실원천이 C#의 `ObservableCollection+bool`에서 **코어(nexa-tree) 가시행 스트림 + OrderedSet 선택**으로 단일화됐다. 현재 골격(ADR-0001~0004)은 **M2~M7을 큰 변경 없이 수용**할 수 있게 예약돼 있다 — **전면 아키텍처 변경은 불필요**(Q3 직답).

그러나 세 갈래로 "지금 손봐야 할 것"이 남았다:

1. **성능(Q1)**: 코어는 네이티브 준함의 토대가 견고하나(10만 노드 벤치 통과), **UI 경계에 동기 I/O·Reset 재실체화 병목**이 남아 대형 폴더 실사용에서 NFR-P를 위협 → **P1/P2/P3가 최우선**.
2. **설계 완결성(Q2)**: "설계 변경 불필요"라고 **완전히 답할 수는 없다** — 실사용 탐색기의 3대 필수(**VFS Provider·nexa-ops·watcher**)가 아키텍처 그림엔 박스로만 있고 **전용 설계·인터페이스 계약이 없다**. M1을 닫기 전 최소 계약 동결 필요.
3. **구조/문서(Q4)**: 설계는 건강하나 **MainWindow god object·`bool left` 이중화·XAML 복붙**(전부 1차 이월)이 C3(교차선택 UI) 얹기 전 **지금이 최저 비용**. 문서 스테일·커밋 추적성은 다시 이월됨.

**Double Commander 비교 결론**: WinUI 3/.NET UI는 유휴 메모리 **2배±**·배포 크기 **3–5배**·콜드스타트 **열위** — "네이티브 준함"이 **UI 계층에는 사실이 아니다**. 단 이는 ADR-0001이 처음부터 감수한 트레이드오프이며 **속도(콜드스타트 제외)는 Rust 핫패스로 방어**된다. **ADR-0001 전면 재검토 트리거는 아니나, NFR-M2/P5 목표치를 스택 현실로 재보정하고 배포 전략을 이원화하지 않으면 "NFR 미달"이 확정**된다.

## Q1. 앱 속도·용량이 네이티브에 준하는 최적화 상태인가?

**부분.** 핫패스(열거·트리 평면화·정렬·필터·선택)는 Rust 코어로 이관돼 10만 노드 expand 5.7ms 등 프레임 예산 내 확인됨. 하지만 **UI 경계**에 구조적 병목이 남아 있다.

### 최우선 성능 액션 (사용자 지정 P1/P2/P3)

### 그 밖의 성능 항목 (나중)

- **[P5] arena 노드 미회수 → 장기 세션 메모리 단조 증가** (`nexa-tree:17` "회수 안 함", collapse가 visible만 제거·노드 보존). 탭 유지한 채 수천 폴더 펼쳤다 접기 반복 시 상한 없음. **앱 측 유휴 RSS 트림·주기 자니터 미구현**(NFR-M3, 설계만). → soak(NFR-R4) 전 필요.
- **[P6] 아이콘 캐시·상한 없음** (`MainWindow:345` `LoadIconAsync`가 행 실체화마다 셸 호출, 확장자/폴더 키 캐시 없음, Reset마다 전량 재발화, 화면 밖 취소 없음). → 확장자·종류 LRU 캐시 + 화면 밖 취소 + 동시 로드 세마포어.
- **[P4] nexa-ops 부재** — 성능 회귀는 아니나(미착수), 착수 시 처음부터 백그라운드 저우선순위 I/O(NFR-N1)·취소·진행률을 코어 설계에 포함해야 재작업 없음. (→ Q2 설계 항목과 통합)
- **[낮음] `Count`가 매 접근 P/Invoke**(`VirtualTreeCollection:112`) — 타이트 루프 진입 시 `int n=Count` 캐시 일관 적용.

### 잘 되어 유지할 것 (성능)
핫패스 코어 이관 완료 · 가상화 지연 실체화 · `nexa_tree_index_of`(4-3, 클릭 실체화 제거) · 탭별 핸들 캐시(4-2) · ABI 레이아웃 가드 · 문자열 수명 규약 · 균일 높이 ChangeView 스크롤(강제 실체화 회피).

## Q1-보강. Double Commander(네이티브) 대비 비교·평가

**대상**: Double Commander(Free Pascal/Lazarus, 완전 네이티브 듀얼 패널) vs Nexa Dir(Rust 코어 + **WinUI 3/.NET8 UI**).

| 항목 | Double Commander | Nexa Dir(예상) | 격차 |
|---|---|---|---|
| UI 툴킷 | 네이티브 컴파일(LCL) | .NET8 + WinUI3(XAML/Composition) | — |
| 콜드스타트 | 즉시(JIT 없음) | CLR+WinAppSDK+XAML 초기화 → NFR-P5(<1s) **위태** | 열위 |
| 유휴 RSS | ~30–60MB | ~60–120MB 추정(관리힙+WinAppSDK+XAML) | **약 2배** |
| 배포 크기 | 다운 ~10MB / 설치 ~40–50MB | self-contained ~150–230MB / fwk-dependent ~33MB+런타임 | **3–5배(디스크)** |
| 포터블 | zip 즉시 | 폴더 zip 가능(무거움), 단일 exe는 COM 활성화 검증 필요 | 열위 |
| 플러그인 | 성숙한 TC호환 생태계(WCX/WDX/WFX/WLX) | WASM+Py/Node(M6, 미구현·생태계 0) | 열위(생태계) |

**핵심 평가**:
- **메모리·크기 격차는 UI 프레임워크 본질** — Rust 코어가 아껴도 유휴 RSS를 지배하는 것은 **관리 CLR 힙 + WindowsAppSDK + XAML 컴포지션**(전부 UI 측)이라 코어가 상쇄 못 함. 사용자 우려가 정확.
- **속도의 90%는 스택 유지로 방어** — 핫패스 Rust화가 실측(10만 5.7ms)으로 검증. **콜드스타트만 예외**이고 R2R로 부분 완화 가능(NativeAOT는 WinUI3 XAML 미지원).
- **ADR-0001 재검토 판정**: **전면 재검토(스택 교체) 불필요**. 재검토 트리거는 "M0 스파이크 실패"인데 성능 스파이크는 통과, **유휴 RSS/트림 soak 실측만 미완**. → 재검토 판단은 **NFR-M2 실측 결과에 유보**. 단 **"UI가 네이티브 준함"이라는 서술은 정정 필요**(유휴 메모리·크기·콜드스타트에서 열위).

**설계/기능 조정 권고**:
1. **[심각] NFR-M2 재보정 또는 soak 실측 선행** — "트림 후 <80MB"는 WinUI3+CLR 상주분 고려 시 비현실적일 수 있음. docs/28 트랙 A(R2R)·B(VMMap 모듈별 private bytes)를 Windows 실측 후, "유휴<150MB / 트림 후 <100–120MB"로 현실화하거나 실측이 통과하면 유지. **추정으로 목표 확정 금지.**
2. **[심각] 배포 전략 이원화** — (a)일반: fwk-dependent MSIX + Windows App Runtime 전제, (b)포터블: self-contained 폴더 zip. 단일 exe는 COM 활성화 제약을 패키징 스파이크로 조기 검증, 실패 시 "포터블=폴더 zip" 공식 폴백. DR-3 "단일 exe"는 검증 전 **비확정** 표기.
3. **[중대] R2R 릴리스 파이프라인 채택**(콜드스타트) + 시작 리소스 지연 로드.
4. **[중대] 상주 규율(NFR-M3/M4) 실제 구현** — 캐시 하드캡+LRU, 유휴 `EmptyWorkingSet` 트림, 메모리 압박 구독. **WinUI3 유휴 열위를 상쇄하는 유일한 실효 레버** → M1 우선순위 상향.
5. **[보통] 플러그인 브리지 검토** — TC호환 읽기전용 WLX/WDX 어댑터로 초기 생태계 확보.
6. **[보통] "네이티브 준함" 서술 정정**(공개 전환 신뢰성).

> **Windows 실측 필요**: Nexa 실제 유휴 RSS, 트림 후 <80MB 달성 여부, R2R 콜드스타트, self-contained 실크기, 단일 exe COM 동작, DC 동일 시나리오 재측정.

## Q2·Q3. 미래 기능 설계 수용성 / 요구 상충 / 아키텍처 변경 필요성

**Q3 직답: 큰 아키텍처 변경은 불필요.** 골격이 M2~M7을 얹도록 예약돼 있고(아키텍처 모듈 표·ADR-0002/3/4), **구조를 바꿀 진짜 상충은 없다**(정책 미정만 존재).
**Q2 직답: "설계 변경 불필요"라고 완전히 답할 수 없다** — 아래 3대 필수 크레이트의 계약이 비어 있다.

### 미래 마일스톤 수용성

| M | 수용 | 리스크 |
|---|---|---|
| M2 미리보기 | **O** | preview 크레이트+ABI만 추가 |
| M3 검색(tantivy/MFT) | **부분→O** | 구조 수용. **index 영속화·권한 ADR 미결**, tantivy 메모리 캡 수치 미정 |
| M4 클라우드/네트워크 | **부분** | **VFS Provider가 스텁, 트리가 로컬 std::fs 직결** → M4 전 열거 경로를 Provider 경유로 리팩터 필요 |
| M5 AI | **O** | 보류 타당(DR-4) |
| M6 플러그인 | **O** | 성숙. 취소/진행률 협조 프로토콜만 미비 |
| M7 라이선스 | **O** | 완결, 잘 격리 |

### 설계 공백 (문서 자체가 없는 것) — 2차 신규

1. **[최우선] nexa-ops 파일 작업 엔진** — 잡큐/트랜잭션/충돌/진행률/취소/Undo. **M1 AC3의 전제인데 설계 문서·크레이트 0**. 선택/드래그/컨텍스트 배선이 이미 붙은 상태라 나중에 정하면 재작업 위험.
2. **[최우선] VFS Provider 통합 계약** — `Provider` trait이 `scheme()`만(`nexa-vfs:72`), 트리가 `read_dir_entries` 직결. `list/stat/read/watch` 노출 + `nexa-tree`가 `&dyn Provider` 경유하도록 M4 전 리팩터. 지금은 인터페이스만 얇게.
3. **[중요] watcher(파일 변경 감시)** — ADR-0004가 명시적 "범위 밖"이나 실사용 필수(외부 변경 반영·선택 pruning). 로드맵상 M1 항목. 트리에 "부모 구간 무효화+선택 정리" 훅 설계 필요.
4. **[중요] 에러 모델 표준** — interop 23함수가 `1/0/-1` 애드혹, `expand`가 IO에러를 `0`으로 뭉갬(빈 폴더 vs 권한거부 구분 불가). 에러 코드 enum + last-error 조회 표준(ops/vfs/preview 공통) → M2/M3 전 동결.
5. **[중요] 취소/진행률 표준** — ops·plugin·search 공통 계약 부재.
6. **[권장] 코어 다소비자 메모리 압박 브로커** — tantivy/wasmtime/썸네일 캐시가 각자 캡만, 전역 압박 조율(NFR-M4) 미설계.

### 인터롭 경계 (2차 신규/이월)
- **[중요] csbindgen 미도입** — 결정(docs/10)은 "C ABI + csbindgen"인데 실제는 **수동 P/Invoke 미러 20개+구조체 3개**. 표면이 프리뷰/검색/ops로 급증하면 미러 드리프트 리스크 선형 증가. 런타임 가드(`VerifyAbi`)는 좋으나 컴파일타임 보장 아님. → M2/M3 전 csbindgen 도입 또는 코드젠.

### 요구 상충 (구조 파괴 없음, 정책 미정만)
- NFR-N1(백그라운드 스로틀) vs NFR-P2(60fps): 임계 SLA 미정.
- NFR-M2(80MB 트림) vs 프리뷰/인덱스 캐시: 개별 캡 합의, **합산 조율 브로커 미설계**.
- 임베디드 터미널 "설계 우선"(05) vs "구현 P2 연기"(03): 문구 정합만 필요.

### 잘 설계되어 유지할 것
코어 트리/선택 모델(ADR-0004, NodeId 기반 교차선택→M4 다중 Provider 안정) · 뷰/패널 확장(ADR-0003 IFileView/IToolPanel) · 재사용 컨트롤(ADR-0002 Nexa.Controls) · 역할분담(코어=핫패스, C#=COM/셸) · 플러그인 격리·capability(docs/09) · 라이선스 격리(docs/17) · ABI 런타임 가드.

## Q4. 기능단위 구조/구현 적합성 · 리팩토링 대상

**설계 훼손 없음 — C1으로 개선.** 진실원천이 코어로 단일화되고 죽은 코드가 정리됨. 다만 아래는 C3(교차선택 UI)·명령 레지스트리를 얹기 전 **지금이 최저 비용**.

### 지금 리팩토링 시급 (이 브랜치)
- **[높음] F1 MainWindow god object + `Nexa.ViewModels` 부재** [1차 A2 이월] — 939줄이 네비/탭/선택/키보드/펼침/아이콘/스크롤/스냅 전담, 순수 로직(`LoadIconAsync/SnapTarget/TabTitle`)까지 포함. net8.0 ViewModels 없어 **C# 테스트 0**. → `PanelViewModel`+`WindowLayoutViewModel` 추출(순수 로직부터).
- **[높음] F2 `bool left` 좌/우 이중화** [1차 C1 이월·악화] — `Navigate/GoUp/ShowTab/UpdateNavButtons` 등 ~20+ 메서드가 `left ? _leftX : _rightX` 관통. → `PanelView{Tabs,ActiveTab,Items,Grid,Header,PathBar,...}` 객체 2개로 치환. F1의 이음새.
- **[중간] F3 XAML 좌/우 패널 ~90줄 복붙** [1차 C3 이월] — 탭바·네비바·행 템플릿 바이트 동일(차이=x:Name/Tag/틴트). → `PanelControl` UserControl + 공유 `ResourceDictionary`. F2와 세트.

### 저위험 퀵윈 (개별 커밋)
- **[중간] F4 interop 죽은 `nexa_dir_*` 제거** [1차 C6 부분] — `nexa_dir_open/next/close`+`DirHandle`가 C# 미사용(자기 테스트만). `NexaEntry`는 레이아웃 가드에만 쓰이나 실소비 없음 → 함께 정리 검토.
- **[낮음] F7 `panic="abort"` 미설정** [1차 B1 이월] — cdylib FFI 패닉 언와인딩 가드. 원라인급.
- **[낮음] F8 심링크 `symlink_metadata`** [1차 B4 이월] — 깨진 링크 attrs=0(숨김 링크 안 숨겨짐) + attrs 유닛테스트.
- **[낮음] D1 STATUS LOC 정정**(721→939) 및 문서 스테일 일괄(아래 문서 섹션).

### 나중 (C3/D 착수 시)
F6 코어 경로기반 펼침(=P3와 통합) · B2 CI P/Invoke 스모크(MSIX 전) · C4 OnGridKeyDown→키맵 레지스트리(명령 팔레트 시) · C5 색상/치수 ResourceDictionary 테마화(설정/테마 시).

### 하지 말 것 (과설계 경계)
- 미사용 선택 API(`SelectAll/ClearSelection/SelectedPaths`) 지금 확장 — C3 실사용 전 최소 유지.
- 코어 `nexa-tree`↔`nexa-vfs`를 지금 Provider 트레잇 주입으로 추상화 — **단, M4 전에는 필요**(Q2 참조). 지금 과설계 vs M4 전 계약의 균형은 "얇은 인터페이스만 먼저".
- 러버밴드/ops를 F1~F3 리팩토링 전에 얹기.

### 잘 구조화되어 유지할 것
코어 단방향 의존(core→vfs→tree→interop) · 진실원천 단일화(C1 성과) · ABI 안전 계층 · 오류 격리(반환코드→try/catch→상태바 + crash.log) · 정직한 문서 규율.
