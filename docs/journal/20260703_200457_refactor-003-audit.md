# refactor/003 통합 감사 리포트 (3차 · 검증용)

> 1차·2차 방식 차용 — 4축 병렬 감사(**설계 부합·구현준비 / 성능 보장 / 배드스멜·구조 / 파일탐색기 FR·NFR**) 통합.
> 분기 기준: `1d9d312`(main, 2차 감사 트랙 A/B 병합 직후). 사용자 요청: 설계 방향 부합·구현준비 충분성 / 성능 보장 / 배드스멜 제거·효율 구조 / 브라우저 FR·NFR 충분성 점검 → **할 일 목록화**(구현은 지시 시).
> **원칙**: 2차에서 완료된 항목(P1/P2/P3/P6·B-2a·B-1) 제외, 미해결·신규만으로 3차 기준 수립.

## 0. 총평

**성능 토대와 코어 아키텍처는 우수하다. 그러나 "탐색기"가 아니라 아직 "뷰어"다.** 두 개의 큰 진실:

1. **[제품] 파일 조작 계층 전무** — 복사/이동/삭제/이름변경/새폴더·컨텍스트 메뉴·클립보드·드래그앤드롭이 **하나도 없다**. 플래그십의 "선택"은 완성됐으나 그 선택으로 **할 일(ops)이 없다**. 이는 M2+ 후순위가 아니라 **현재 마일스톤(M1)의 P0 공백**이며, 탐색기 대체라는 제품 정체성의 최소 요건이다.

2. **[성능] 렌더는 가상화됐으나 데이터 계층(`_cache`)은 무한 성장** — P1/P2/P3/P6은 "첫 진입·펼침 1회"의 블로킹을 실제로 없앴지만, `VirtualTreeCollection._cache`가 뷰포트 이탈 시 축출되지 않아 **선택/포커스/토글이 O(방문 행수)로 열화**하고 다중 탭·장기 세션에 **메모리 상한이 없다**. 즉 **NFR-M(메모리 상한)·NFR-P2(장기 60fps)가 실사용에서 보장되지 않는다** — 성능 작업을 했음에도.

그 밖에: **설계 계약 공백**(에러 모델·VFS Provider·watcher)이 M1 마감과 M2+ 전체의 관문으로 재확인됐고, **인프라 부채**(설정이 인메모리 전용→재시작 소실, i18n 0·한국어 하드코딩 163+, 접근성 0)가 상용/공개 전환 전에 급증하기 전 지금이 최저비용이다. 구조적으로는 죽은 코드·추출 가능한 순수 로직·XAML 복붙이 남아 있으나 **설계 훼손은 없다**.

## 축1 · 설계 방향 부합 + 구현/준비 충분성

**설계 훼손 없음. 문제는 "공백의 성격"** — 일부 공백은 "M4 이후"가 아니라 **M1 완료 자체를 막는 차단자**다.

- **[중대] nexa-ops 부재** — `Tree::selected_paths()`가 작업 엔진 입력 계약을 이미 노출하는데 소비자가 없음. M1 완료 기준·플래그십 AC3/AC4의 절반이 공백. (→ 트랙 B)
- **[중대] VFS Provider 계약이 비어 있고 트리가 우회** — `Provider`가 `scheme()`만, 트리가 `read_dir_entries` 자유함수 직결. **M4·M6의 핵심 차단자**, 로컬만인 지금이 재배선 최저비용. (→ C-2)
- **[중대] 에러 모델 1/0/-1 애드혹** — IO 오류를 `0`으로 뭉개 "왜 실패했는지" 소실 → 에러 행·재시도 UX 불가. ABI 표면 커질수록 부채 가속. (→ C-1)
- **[중대] watcher 완전 부재** — 외부 변경 미반영, stale 경로가 ops로 갈 위험. arena append-only가 향후 pruning 재설계를 압박(준비도 낮음). (→ C-3)
- **[중간] csbindgen 미도입** — `VerifyAbi`는 레이아웃만, 함수 시그니처 드리프트 사각. 표면 커지기 전 도입이 최저비용. (→ E-5)
- **[중간] docs/01 아키텍처 스테일** — 모듈표에 **nexa-tree 누락**, 미존재 크레이트(index/preview/ops/ai)를 현존처럼 표기. (→ C-4)
- **[중간] 순수 로직 여전히 MainWindow 집중(933줄)** — MVVM 설계 대비 부분 배치, 테스트 불가 UI 로직 축적. (→ E-2)
- **[낮음] 빌드 취약**(Directory.Build.props 부재·TFM 3중복·core 경로 하드코딩). (→ E-7)

**M2~M7 수용성**: M5·M7은 지금 골격이 무리 없이 수용. **M2·M3는 에러 모델·Provider 선결. M4·M6는 Provider 계약이 열리지 않으면 골격 변경 없이는 불가** → 트랙 C가 M2 이후 전체의 관문.

## 축2 · 성능 보장 충분성

**"첫 진입·펼침 1회" 블로킹은 실제 해소. 그러나 가상화가 렌더에만 있고 데이터 계층은 무경계** → NFR-M1·NFR-P2 미보장. 핵심 병목은 하나의 뿌리(`_cache` 무경계)에서 파생.

- **[높음] `VirtualTreeCollection._cache` 무한 증가** — `this[index]`가 방문 행을 전부 캐시하고 뷰포트 이탈 축출 없음(`Close`/`AdoptHandle`만 비움). 파생 열화 3종:
  - `RefreshSelectionVisuals`(선택마다 `foreach(_cache) TreeIsSelected` P/Invoke) → **키보드 ↑/↓ 한 번이 방문 행수만큼 네이티브 왕복**.
  - `ApplyDiff`(토글마다 `_cache` 전체 Dict 재구축).
  - `SetPanelFocused`(포커스 전환마다 전체 순회).
  → 큰 폴더를 키보드로 내려갈수록 키당 비용 선형 증가. **NFR-M1/P2/P** 위반. (→ A-1)
- **[높음] 다중 탭 총량 상한 없음** — 비활성 탭도 핸들 유지(arena + `_cache`), 닫을 때만 회수. 10탭×대형 폴더 = arena 10 + 캐시 10 무캡. NFR-M2/R4. (→ A-2)
- **[중간] 코어 arena 노드 미회수(P5)** — `enumerate`는 push만, `collapse`는 visible만 제거. 동일 트리에서 펼침/접힘 반복 시 세션 내 단조 증가(네비게이션은 Close로 회수됨=양호). (→ A-3)
- **[중간] 유휴/최소화 트림·자니터 전무** — `EmptyWorkingSet`·압박 구독·주기 타이머 0건. 위 두 건과 결합해 RSS 단조 상승. NFR-M2/M3/M4. (→ A-4)
- **[중간] 아이콘 큐 처리율 낮음** — 동시 4·80ms·완료 즉시 리필 안 함 → 실효 ~50아이콘/초, 정착 후 한 화면 ~0.8s. 첫 배치도 최대 80ms 대기. (→ A-5)
- **[중간] `ScrollIndexIntoView` 동기 `UpdateLayout`×최대10 재시도** — 깊은 인덱스 상향 네비 시 강제 레이아웃 몰림. (→ A-6)
- **[낮음] `Count` 매 접근 P/Invoke, `CopyTo`/enumerator 전량 실체화**(오용 시 가상화 붕괴). (→ E-2/A-6)

**측정 필요(실측 0)**: soak RSS·핸들 누수 / 콜드스타트 <1s / 빠른 스크롤+아이콘 60fps / 대형 단일 펼침 Add 비용 / `_cache` 성장의 실효 비용. (→ A-7)

## 축4 · 파일 탐색기 FR/NFR 충분성

**표준(Explorer + Total/Double Commander + Path Finder) 대비 핵심 공백은 "탐색 이후의 조작·영속·접근성" 계층에 집중.** 검색·미리보기·클라우드·플러그인의 "없음"은 로드맵상 의도적 후순위(시급 아님)이나, **아래 Top 10은 대부분 M1 범위인데 미구현/설계만**이다.

### 가장 시급한 갭 Top 10 (M1 범위 실질 공백)
1. **파일 작업 엔진**(복사/이동/삭제/이름변경/새폴더, `nexa-ops`) — 탐색만 되고 조작 불가 = "뷰어". [B-1]
2. **컨텍스트 메뉴**(셸 `IContextMenu` + 고유) — 우클릭 자체 없음. [B-2]
3. **클립보드**(Ctrl+C/X/V) — 테이블 스테이크인데 요구 문서에도 항목 부재(사각지대). [B-3]
4. **드래그앤드롭**(패널간·경로바·런처) + **러버밴드 선택** + Ctrl+A/Home/End/PageUp. [B-4]
5. **파괴적 작업 안전장치**(휴지통·Undo·충돌 정책·진행률/취소) — ops와 동반 설계 필수(나중에 얹으면 트랜잭션 재설계). [B-1 동반]
6. **교차폴더 다중 선택 완성**(C3/C4) — 플래그십(시그니처)의 절반, F11은 패널 내 선택뿐. [B-5]
7. **설정 영속화(JSON) + 세션/창 복원** — 현재 **인메모리 전용**, 재시작 시 설정·탭·펼침·창위치 전부 소실. [D-1]
8. **사이드바**(즐겨찾기/드라이브/최근) — 표준 탐색 시작점 부재. [B-7]
9. **i18n 인프라** — `ResourceLoader`/`.resw` 0, 한국어 하드코딩 163+. 문자열 쌓일수록 재작업 기하급수, 공개(DR-5) 전 지금이 최저비용. [D-2]
10. **접근성**(AutomationProperties/UIA Peer) — 0건. 커스텀 컨트롤이 표준 대체 → 스크린리더 전무. 컨트롤 늘기 전 배선. [D-3]

### NFR 요약
- **충족**: NFR-M1(가상화·단 `_cache` 예외=축2), NFR-Q1(테스트/CI 코어21+C#25). 
- **부분**: P1/P2(측정 미완), R3/R5(핵심 경로만), N1(오프로드만·우선순위 없음).
- **미구현/위험**: **NFR-M2/M3/M4**(트림·자니터·압박) · **접근성 A1**(스크린리더 0) · **i18n**(0) · **세션/설정 영속**(0) · NFR-R1/R2(파일작업 부재라 미해당→착수 시 최우선 설계).

## 종합 결론 · 권고 착수 순서

- **지금 손볼 3대**: ① **트랙 B 핵심 기능**(제품이 "탐색기"가 되려면 필수 — B-1 ops + B-2 컨텍스트메뉴 + B-3 클립보드) ② **트랙 A 성능·메모리 보증**(A-1 `_cache` 상한 — 이미 한 성능 작업을 실보장으로) ③ **트랙 C 설계 계약**(C-1 에러·C-2 Provider — B와 M2+의 전제).
- **동시성**: B-1(ops)은 **C-1(에러 모델)·A-1(선택 성능)**과 맞물림 — ops 착수 시 에러/취소/진행률 표준을 함께 동결해야 재작업 없음.
- **저비용 병행**: 트랙 E(죽은 코드·순수 추출)와 D-2(i18n)는 문자열/코드가 더 쌓이기 전 **지금이 최저비용** — 큰 기능 착수와 별개로 틈틈이.
- **자기인식 정확**: 트랙 C 공백은 2차에서 이미 식별됨. 3차의 새 발견은 **① 성능이 "실보장"이 아니라는 점(_cache)** **② 파일 조작 계층 전무가 M1 P0라는 점** **③ 영속/i18n/접근성 인프라 부채**다.
- **하지 말 것**: watcher/preview/search를 ops·Provider·에러 모델 계약 동결 전에 얹기 · `_cache` 상한 없이 새 대형 목록 기능 추가.
