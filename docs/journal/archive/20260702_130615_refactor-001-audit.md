## 총평

규율은 좋다 — 문서가 자기 한계를 **정직하게** 표기(‘초안’·‘다음 단위 C1’)하고, CI는 green(fmt/clippy/test/deny/app). 그래서 최대 아키텍처 괴리는 **은폐가 아니라 관리되는 드리프트**다. 다만 위험은 “버그”가 아니라 **구조**에 있다:

- **한 줄 결론**: 지금 고치면 싼 것 = **① 코어 VisibleRow 스트림(C1) + ② `Nexa.ViewModels` 추출**. C3(교차선택)·C4(ops)·D(명령 레지스트리)를 지금의 `ObservableCollection<DirItem>`+bool 선택 위에 더 쌓기 전에 잡아야 한다. 미루면 재작업 비용이 계속 커진다.

## A. 최우선 — 구조·정합성 (지금 안 잡으면 비싸짐)

### A1. 플래그십 트리/선택이 C# 코드비하인드에 축적 · O(n) · NFR(10만/60fps) 위배
- 설계: `docs/01 §3`(핫패스=코어), `docs/07 §3-4`(코어 `VisibleRow` 평면 스트림 + `OrderedSet<NodeId>` 선택, AC5=10만 노드 <150ms/60fps), `docs/10 §2`.
- 현실: `MainWindow.xaml.cs`의 `ExpandInPlace`(L431)·`CollapseInPlace`(L454)·`ApplySavedExpansion`(L468), 선택은 `DirItem.IsSelected` bool, 정렬·가시성 필터는 `NativeInterop.ReadDir/SortItems/IsVisible`(L255–316). 코어는 `nexa_dir_open/next/close`만 노출 — expand/visible-row/selection API 없음.
- 문제: 선택 해제 `foreach … IsSelected=false`(L524/544/961/974), `ApplySavedExpansion` 로드마다 재스캔, `SelectByPath`/`ParentIndex` 선형 스캔 → **UI 스레드 O(n)**. 확장은 **동기 재귀 `ReadDir`**(L440)로 지연/스트리밍 창 없음. 수백 행엔 OK, AC5와 정면 충돌.
- 상태: `docs/19` F10/F11·`STATUS §6`(C1)에 **명시적으로 인지됨** → 관리되는 드리프트. 그러나 가장 하중이 큰 괴리.
- 권고: **C1(코어 VisibleRow + expand/collapse + 선택 OrderedSet)을 다음 비선택 단위로.** 착수 전 현재 C# 경로를 10만 행으로 벤치해 NFR 격차를 수치화하고 UI의 추가 결합을 차단.

### A2. `Nexa.ViewModels` 부재 — 로직 전부 1124줄 코드비하인드(맥 테스트 불가, MVVM 문서와 배치)
- 설계: `docs/01 §2/§6`(MVVM, `app/Nexa.ViewModels/`·`app/Nexa.Interop/`), `docs/11 §6-1`(net8.0 분리→맥 빌드/테스트).
- 현실: `app/`엔 `Nexa.App`·`Nexa.Controls`뿐. ViewModels 없음, P/Invoke는 `Nexa.App/NativeInterop.cs` 인라인. 네비/선택/탭/확장/스냅 전부 `MainWindow.xaml.cs`(**1124줄**). C# 테스트 0개.
- 권고: 네비/선택/탭/트리 로직을 `Nexa.ViewModels`(net8.0)+단위테스트로 추출. 모델이 ~1k줄인 **지금이 가장 쌈**(C3/C4/D가 얹히기 전).

### A3. ABI 버전이 장식 — 검사 안 함 · 구조체 레이아웃 가드 없음 · csbindgen 미사용(동결 결정과 불일치)
- 현실: `MainWindow.xaml.cs:87-89`가 `nexa_abi_version()`을 **표시만**. 비교/거부 없음(주석·`interop:14`은 “불일치 시 로드 거부”라 주장). v1→v2 범프(F24)는 **사실상 무의미** — 구형 dll이 v2로 읽혀 `attrs`/`size` 오염돼도 무오류.
- 현실2: `NexaEntry` `#[repr(C)]`(Rust)↔`[StructLayout(Sequential)]`(C#) 바이트 동치가 **강제되지 않음**(x64에선 우연히 일치, 재정렬/arm64서 조용히 깨짐). csbindgen(DR-1 동결)은 미사용, 수기 미러.
- 권고: 로드 시 `nexa_abi_version()==Expected` 검사→불일치면 네이티브 경로 차단. `size_of::<NexaEntry>()`/오프셋 테스트 + C# `Marshal.SizeOf` 가드(또는 `nexa_entry_size()` export). 장기: **csbindgen 도입 또는 “보류” ADR/journal 명시**(동결 결정과 현실 불일치 해소).

## B. 중요 — 안전성·품질 (코어/CI)

- **B1. FFI 패닉 언와인딩 위험**: `nexa_dir_next` 경로에 `panic="abort"`/`catch_unwind` 없음(`nexa_poc_add`은 wrapping으로 회피했으나 열거 경로가 재도입). cdylib 프로파일 `panic="abort"` 또는 본문 `catch_unwind`→`-1`. *(core #3, 원라인급)*
- **B2. CI가 P/Invoke 왕복 미검증**: `app` job은 `dotnet build`만 — dll 로드/마샬링/ReadDir 미실행. 헤드리스 스모크(`Add-Type` 또는 `Nexa.App.Tests`: `abi==2`·`poc_add(2,3)==5`·temp `ReadDir`) 추가. *(core #4)*
- **B3. 핵심 로직 무테스트**: Windows `attrs`(숨김 필터 근간, `#[cfg(windows)] file_attrs`)·심링크 매핑·interior-NUL·스트리밍 지연성·`modified=-1` 센티널 무테스트. 숨김 필터 `IsVisible`는 **UI 어셈블리에만** 존재→맥 테스트 불가. *(core #5/#7/#8)*
- **B4. 심링크 metadata follow 의미 혼재**: `kind`는 미추종(`file_type`), `size/modified/attrs`는 추종(`metadata()`) → 깨진 심링크서 `attrs=0`→숨김 링크가 안 숨겨짐. `symlink_metadata()` 사용 또는 의미 문서화. *(core #5)*
- **B5. 빌드 취약**: `Nexa.App.csproj`가 `..\..\core` 하드코딩·`cargo` 존재 체크 없음·**arm64 크로스 미비**(csproj는 win-arm64 광고하나 cargo는 host-arch만 빌드). TFM은 두 csproj 리터럴 중복→`Directory.Build.props`로 공유(CS0234 재발 방지). *(core #9/#10)*
- **B6. cargo-deny 게이트 미검증**: 의존성 0개라 라이선스 거부가 end-to-end 미증명. 무거운 deps(tantivy/wasmtime) 전에 GPL 크레이트로 거부 확인 + advisory severity 명시. *(core #6)*
