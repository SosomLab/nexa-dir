# 17 · 라이선스 정품 인증 (오프라인 1차 / 온라인 2차)

> 유료 라이선스(DR-5, [13](13-licensing.md)) **집행 메커니즘**. 프로그램에 **라이선스 키/인증값**을 입력해
> 정품 인증 → 기능/사용기간 잠금 해제. **1차 오프라인 인증**, **2차 온라인 인증**으로 단계 도입.
> 본 문서는 **기술 모델 추천 + 절차 설계 + 개발 TODO**.

---

## 1. 핵심 기술 모델 추천 (트렌드 · 안정성)

> 결론: **비대칭 서명(공개키 암호) 기반 라이선스 토큰** — 클라이언트엔 *공개키만* 두어 **오프라인 검증**,
> 위조 불가. 알고리즘은 **Ed25519**(현 표준·빠름·64바이트 서명), 토큰 포맷은 **PASETO v4.public**(JWT의 안전한 대안)
> 또는 서명된 CBOR. 온라인 단계는 **라이선스 서버(Cloudflare Workers + D1/KV)** 로 확장.

| 항목 | 추천 | 이유 |
| --- | --- | --- |
| 서명 알고리즘 | **Ed25519** | 모던·고속·작은 서명, Rust `ed25519-dalek`/.NET `NSec` 지원 |
| 토큰 포맷 | **PASETO v4.public** (대안 JWT/EdDSA, 서명 CBOR) | 위조·혼동공격 방지, 자기완결 |
| 클라이언트 검증 | **공개키만 내장** | 바이너리 유출돼도 키 위조 불가(대칭키/시리얼 알고리즘과 차별) |
| 머신 바인딩 | **하드웨어 지문(node-lock, 선택)** | 키 공유 방지. 과도 바인딩은 지양(2-of-3 관용) |
| 온라인 서버 | **Cloudflare Workers + D1/KV** | 좌석/실효/갱신/통계, 보유 스킬과 정합 |
| 빌드 vs 구매 | 오프라인 서명기는 **자체 개발**, 온라인은 **Keygen(오픈소스, 셀프호스트)** 평가 | 비용·통제 균형 |

> ❌ 피해야 할 것: 대칭키/난독화된 "시리얼 알고리즘"(바이너리 리버싱 시 키 생성기 복제 가능),
> 클라이언트에 **비밀키 내장**(절대 금지).

## 2. 구성요소

```
┌─────────────────────────────┐         ┌──────────────────────────────┐
│  라이선스 생성기 (별도 툴)   │  서명   │  Nexa Dir 앱 (클라이언트)     │
│  - 비밀키 보관(오프라인/HSM) │ ──토큰─▶│  - 공개키 내장               │
│  - 사용자 정보 입력          │         │  - 토큰 검증(서명/만료/지문) │
│  - 인증값(해시/토큰) 생성    │         │  - 기능/기간 잠금 해제       │
└─────────────────────────────┘         └──────────────────────────────┘
        (2차) 온라인: Cloudflare Workers 라이선스 서버 — 발급/검증/좌석/실효/갱신
```

- **라이선스 생성기(license generator)** = "별도 인증 프로그램". **비밀키를 보유**(저장소·앱에 절대 미포함,
  시크릿/HSM/오프라인 머신). 운영자가 사용자 정보·엔타이틀먼트를 입력 → 서명 토큰 산출.
- **앱(클라이언트)** = 공개키로 토큰 검증. 비밀 없음.
- **온라인 서버(2차)** = 발급/검증/좌석관리/실효(폐기)/구독 갱신.

## 3. 인증값(토큰)에 담기는 정보

> 사용자가 입력한 **사용자 정보 + 키를 1개의 토큰(해시·서명)** 으로 만든다.

```
payload = {
  lic_id,              // 라이선스 고유 id
  product: "nexa-dir",
  edition,             // 에디션/티어
  features: [...],     // 잠금 해제 기능 플래그
  user: { name, email or org },   // 사용자 정보(서명에 포함)
  machine_fp?,         // (선택) 하드웨어 지문 — node-lock
  seats,               // 좌석 수
  issued_at,
  expires_at?          // (선택) 사용 기간(구독)
}
token = base64url( payload || Ed25519_sign(privkey, payload) )   // = 인증값(단일 문자열)
```

- 자기완결 토큰: 앱은 토큰만으로 사용자 정보·엔타이틀먼트·서명을 모두 확인(별도 정보 재입력 불필요).
- 대안(짧은 키): "이름 + 시리얼" 형태 — 시리얼 = `Ed25519_sign(이름+제품id)`. 앱에 이름+시리얼 입력해 검증.
- **단일 해시 전달**: 위 토큰 문자열 자체가 "사용자 정보와 키를 합친 1개의 값".

## 4. 인증 절차 (요구안 반영)

### 4-1. 1차 — 오프라인 인증
```
1) (결제 선행 가능) 사용자에게 필요한 정보 요청:
   - 필수: 이름/이메일(또는 조직), 에디션/구매 항목
   - node-lock 시: 앱이 표시하는 "머신 지문" 문자열
2) 라이선스 생성기가 입력 정보로 서명 토큰(인증값) 생성  → 사용자에게 전달
3) 사용자가 앱에 인증값 입력 → 앱이 공개키로 검증:
   - 서명 유효 + 제품/에디션 일치 + (만료 미도래) + (지문 일치) → 잠금 해제
   - 기능 잠금 / 사용 기간 잠금 해제, 결과를 안전 저장(변조 감지)
```
- 장점: 인터넷 불필요, 폐쇄망 가능. 단점: 실효(폐기)·좌석 관리가 약함 → 2차 온라인 보완.
- 기간 만료(구독): `expires_at` 검사 + **시계 롤백 탐지**(마지막 사용시각 저장, 과거로 되돌리면 차단).

### 4-2. 2차 — 온라인 인증 (활성화/검증)
```
1) 앱이 라이선스 키를 서버에 활성화 요청(+머신 지문)
2) 서버: 좌석 수 확인·기기 등록·실효 여부 확인 → 단기 서명 엔타이틀먼트(JWT/PASETO) 발급
3) 앱: 응답 검증 후 캐시(오프라인 유예 기간) → 주기적 재검증(heartbeat)
```
- 추가 기능: **좌석/동시활성 제한, 키 폐기(환불·악용), 기기 이전(deactivate), 구독 자동 갱신, 사용 통계**.
- 서버: **Cloudflare Workers + D1(키/활성화 기록) + KV(캐시)**, 응답 Ed25519 서명, 비밀키는 서버 시크릿.

## 5. 우리 아키텍처 매핑

| 구성 | 위치 / 기술 |
| --- | --- |
| 클라이언트 검증 | **Rust 코어 `nexa-license` 크레이트** — `ed25519-dalek`로 토큰 검증, 만료/지문/기능 판정, 공개키 내장 |
| 엔타이틀먼트 상태 | 코어 → 인터롭으로 UI에 노출(기능 게이팅), 변조 감지 저장(서명+OS 보호 경로) |
| 머신 지문 | 코어 `fingerprint` — Windows MachineGuid + 볼륨시리얼 + CPU 등 조합 해시(2-of-3 관용) |
| 라이선스 생성기 | **별도 도구**(권장: 별도 **private 저장소** `nexa-license-gen`, Rust CLI→후속 GUI). 비밀키 보유 |
| 온라인 서버(2차) | **Cloudflare Workers + D1/KV** (별도 배포), 발급/검증/좌석/실효 |
| 비밀키 보관 | 저장소·앱 **미포함**. 오프라인 머신/HSM/시크릿 매니저. 유출 시 키 로테이션 절차 |

## 6. 보안·운영 고려

- **클라이언트엔 공개키만** — 비밀키는 어디에도 배포 금지(유출 시 전체 위조 가능).
- **리버싱 방어는 한계**(네이티브라도 검증 우회 패치는 가능) → 핵심은 *위조 불가 + 온라인 실효*. 과도한 난독화보다
  합법 사용자 경험 우선.
- **키 로테이션**: 공개키 버전 필드 → 앱이 복수 공개키 지원, 침해 시 신키 발급/구키 폐기.
- **프라이버시**: 사용자 정보 최소 수집, 토큰/지문 처리 고지(상주 규율·보안 NFR과 일관).
- **결제 연동(선행)**: 스토어/결제 후 생성기 발급 자동화(2차에서 서버 연동).

## 7. 단계적 개발 (요약 — TODO는 §8)

- **A. 오프라인 1차 (MVP)**: Ed25519 키쌍 발급 → `nexa-license`(검증) + `nexa-license-gen`(서명 CLI) →
  앱 인증 UI(키 입력/상태) → 기능·기간 게이팅 → 머신 지문(선택).
- **B. 온라인 2차**: Cloudflare Workers 라이선스 서버(D1/KV) → 활성화/검증/좌석/실효 → 앱 온라인 활성화 + 오프라인 유예.
- **C. 운영**: 결제 연동, 키 로테이션, 통계/대시보드, 환불·이전 절차.
