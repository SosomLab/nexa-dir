> 일괄 이름변경([25 §6](25-bulk-rename.md))의 **개발자 확장**: 사용자가 **Python**으로 작성한 함수를 리네임
> 파이프라인에서 호출한다. 사용자 요청(2026-07-08): "사용자 정의 함수는 Python으로". 본 문서는 **기술 구조 +

## 1. 요구 & 제약 (왜 "그냥 CPython 임베드"가 아닌가)

리네임 UDF는 **수천 파일에 반복 호출**되는 **순수 변환**(이름+메타 → 새 이름)이다. 요구:

| 요구 | 의미 |
|---|---|
| **진짜 Python** | 사용자가 익숙한 Python 문법·의미로 함수 작성 |
| **샌드박스** | 이름변환기가 FS/네트워크/프로세스에 접근하면 안 됨(악성/실수 스크립트 방어) |
| **결정성** | 같은 입력 → 같은 출력(미리보기 = 실제 적용). 시계·난수·환경 의존 배제 |
| **배치 성능** | 파일당 호출이 싸야 함(파싱 1회·인터프리터 재사용·병렬) |
| **오류 격리** | 한 파일 예외/무한루프가 배치·앱을 죽이지 않음 |
| **포터블 배포** | 무패키지(unpackaged) 배포 — 무거운 런타임 번들·C 의존 회피 우선 |

**결론(2단 구조)**: 요구를 한 엔진으로 다 만족하는 Python은 없다 → **2계층**.
- **Tier A(기본)**: **RustPython**(순수 Rust Python 인터프리터) 임베드 — 샌드박스·결정성·무런타임번들 모두 충족. 이름변환에 충분.
- **Tier B(옵션·파워유저)**: **아웃오브프로세스 CPython** — 사용자의 Python + 서드파티 패키지 필요 시. 프로세스 격리로 크래시/행 방어.

> (참고: 앞서 [25 §6]은 결정성·샌드박스 이유로 Starlark를 권했으나, **Python 자체가 요구**이므로 RustPython로 그 목표를
> 동일하게 달성한다. Starlark/Rhai는 Python 비요구 시의 대안으로 보류.)

## 2. 계층 비교

| | **Tier A · RustPython(임베드)** | **Tier B · CPython(아웃오브프로세스)** |
|---|---|---|
| 문법 | Python 3(순수 Rust VM) | Python 3(사용자 설치 CPython) |
| 배포 | **코어 바이너리에 정적 링크**(런타임 번들 0) | 사용자 Python 필요(자동 탐지) |
| 샌드박스 | **★ 강**(빌트인·import 화이트리스트·무 ambient) | ○ 중(프로세스 경계 + Job Object 제한, 완전격리는 불가) |
| 결정성 | **★**(시계/난수/환경 미노출) | △(full Python — 보장 불가 → 적용 전 재실행) |
| 성능 | ○ 파일당 인터프리터 호출(가벼움, 병렬) | ○ 워커 1개 + 배치 1왕복(프로세스/RPC 상각) |
| stdlib | 핵심만(문자열·re·datetime 포맷) | **전체** + pip 패키지 |
| 크레이트 | `rustpython-vm` | subprocess + `nexa-udf-host` 프로토콜(=M6 T2 RPC) |
| 맥 개발 | **★ 순수 Rust → 맥 단위테스트** | 맥에서도 동작(사용자 python3) |

**권장 기본 = Tier A**. Tier B는 "패키지 필요" 사용자에게만 opt-in(대화상자에서 엔진 선택).

## 3. 함수 계약 (사용자가 작성하는 것)

`~/AppData/Roaming/NexaDir/rename-udf/*.py` 에 파일을 둔다. 각 파일은 하나 이상의 함수를 선언:

```python
# myrules.py — 촬영일 접두 + 원본, 카메라 없으면 스킵
def photo_prefix(ctx):
    date = ctx.meta.get("DateTaken") or ctx.date["modified"]   # 널 처리
    return f"{date[:10]}_{ctx.name}"

def track_name(ctx):
    n = str(ctx.meta.get("Track", "")).zfill(2)
    return f"{n} {ctx.meta.get('Title', ctx.name)}"
```

### `ctx` (읽기 전용 컨텍스트 — 호스트가 계산해 주입)
| 필드 | 내용 |
|---|---|
| `ctx.name` / `ctx.ext` | 확장자 제외 이름 / 확장자(점 없이) |
| `ctx.stem` / `ctx.path` / `ctx.parent` | 전체 이름 / 전체 경로 / 상위 폴더명 |
| `ctx.index` / `ctx.count` | 1-기준 순번 / 총 개수(연번용) |
| `ctx.meta` | dict — EXIF/IPTC/ID3/PDF… (`nexa-meta`, [25 §5]) |
| `ctx.date` | dict — `modified`/`created`/`accessed`(ISO 8601 문자열) |

- **반환**: 새 이름 문자열(확장자 포함 여부는 대화상자 옵션). 빈/`None`/예외 → **널 정책**([25 §4-3]: 대체/스킵/경고).
- **순수 함수**: 입력=ctx만. 시계·난수·환경·FS 접근 없음(있어도 샌드박스가 차단). → 미리보기=적용 결과 동일.
- **헬퍼(무 import)**: `left/right/substr/pad/proper/…`([25 §4-2])를 내장 노출(선택) — 문자열 작업 편의.

## 4. 동작 메커니즘 — 배치 실행 파이프라인

```
[선택 N개] → (호스트) 항목별 ctx 계산(index·date·meta)
          → (엔진) rename(ctx) 호출  ──Tier A: 인프로세스 VM
                                     └─Tier B: 워커에 배치 전송(1왕복)
          → (원본→새이름) 목록 → 미리보기 표(충돌 검출) → 적용(nexa-ops 트랜잭션·Undo)
```

### 4-1. 로드 & 컴파일(1회)
- 파일 선택 시 **1회 파싱·컴파일**(RustPython: AST/바이트코드 / CPython: 워커가 `import`) → **모든 항목에 재사용**(파일당 재파싱 없음).
- 함수 목록을 대화상자 "UDF" 드롭다운에 노출. **핫리로드**: 파일 변경 감지(watcher) → 재컴파일.

### 4-2. 항목별 호출 (결정성의 핵심)
- 호스트가 **index·date·meta를 계산해 `ctx`에 주입** → 엔진은 시계/난수/환경을 못 봄 → **재실행해도 동일**.
- 순수 함수라 **병렬 가능**(Tier A: 인터프리터 인스턴스 여러 개, 공유 상태 없음).

### 4-3. 미리보기 = 드라이런
- UDF는 항상 **부수효과 없이** 이름만 계산 → 미리보기 표(`원본 → 새 이름`)에 즉시 반영, **충돌 하이라이트**(중복·빈 이름·금지문자·기존 파일).
- Tier B(비결정 가능성)는 **적용 직전 1회 재실행**해 미리보기와 diff 검증(불일치 시 경고).

### 4-4. 적용
- 확정된 (경로, 새 이름) 목록 → `nexa-ops` 리네임(충돌 정책·진행률) → **하나의 트랜잭션**(B-13u Undo).

## 5. 샌드박스 & 격리 (요구: 보안·결정성·오류격리)

### Tier A · RustPython
- **빌트인 화이트리스트**: 순수 함수만 노출. **제거**: `open`·`__import__`·`eval`·`exec`·`input`·파일/OS/네트워크.
- **import 정책**: 기본 차단, **큐레이트 허용**(`re`, `datetime`는 **포맷 전용 심**만 — 현재시각 API 미노출).
- **자원 상한**: **명령 스텝 예산(fuel)** 초과 → 그 항목만 오류(무한 루프 차단), **메모리 캡**. 배치는 계속.
- **read-only ctx**: `meta`/`date`는 frozen 매핑. 핸들(파일/소켓) 없음.
- **패닉 경계**: 각 호출을 `catch_unwind`로 감싸 VM 패닉이 프로세스를 죽이지 않게.

### Tier B · CPython(아웃오브프로세스)
- **격리 실행**: `python -I -S`(isolated: site 무시·env 무시) 장수명 워커 1개.
- **RPC**: stdio 길이-접두 JSON/msgpack. 호스트 `{fn, items:[ctx…]}` → 워커가 파일 1회 import·`map(rename, items)` → 결과 배열. **배치 1왕복**(프로세스/직렬화 비용 상각).
- **OS 샌드박스(Windows)**: **Job Object**(CPU·메모리 상한·자식 프로세스 금지·kill-on-close), 선택적 제한 토큰/AppContainer, 네트워크 정책 차단(best-effort). *full Python은 완전 샌드박스 불가 → "격리된 신뢰" 계층으로 명시.*
- **타임아웃/행**: 배치·항목 타임아웃 → 워커 kill + 해당 항목 오류. 크래시 → 워커 재시작·배치 실패 우아 처리.
- 이 계층은 **M6 플러그인 T2(아웃오브프로세스 RPC, [09](09-plugin-architecture.md))** 인프라 재사용.

### 공통 — 오류 격리
- 항목별 try/except 경계 → 한 항목 예외/타임아웃은 **그 행만 "오류"** 표시(적용 제외), 나머지 진행.

## 8. 단계적 구현
- **1) 계약·러너(Tier A)**: RustPython 임베드 + `ctx` 주입 + 샌드박스(빌트인 화이트리스트·fuel) + 함수 로드/호출. 맥 단위테스트(순수).
- **2) UI 통합**: 대화상자 "Python 함수" 블록·드롭다운·핫리로드·미리보기·오류행.
- **3) Tier B(옵션)**: 아웃오브프로세스 CPython 워커 + RPC + Job Object 격리 + 재실행 검증(=M6 RPC와 공유).
- **4) 헬퍼·토큰**: 내장 문자열 헬퍼 노출·`{fn:...}` 토큰·프리셋 저장.

## 9-1. Starlark vs Python — 설치·크기·성능 비교 (사용자 질문 2026-07-08)

세 후보를 **"별도 설치 필요 · 실행 파일 크기 영향 · 성능"** 축으로 비교. (RustPython/Starlark는 코어 `nexa-interop.dll`에 정적 링크.)

| 항목 | **Starlark** (starlark-rust) | **RustPython** (임베드) | **CPython** (아웃오브프로세스) |
|---|---|---|---|
| 문법 | 파이썬 **유사**(부분집합) | **진짜 Python 3** | **진짜 Python 3(전체)** |
| **별도 설치** | **불요**(정적 링크) | **불요**(정적 링크) | **필요**(사용자 python3) |
| **바이너리 크기 영향** | **+1~3MB**(가벼움) | **+8~15MB**(VM+파서+일부 stdlib) | **~0**(런타임 미포함) |
| **성능(파일당 호출)** | **★ 최속**(경량 VM, fuel) | ○ 양호(순수 Rust VM — CPython보다 느리나 문자열 변환엔 충분) | ○ 배치 1왕복이면 양호, 소량 반복이면 프로세스/IPC 비용 |
| 샌드박스 | ★★(설계상 하메틱) | ★(빌트인/​import 화이트리스트) | 중(프로세스 격리, 완전격리 불가) |
| 결정성 | ★★ | ★ | △ |
| stdlib/패키지 | 없음(순수) | 핵심만 | **전체 + pip** |
| 맥 단위테스트 | ★ 순수 Rust | ★ 순수 Rust | ○(python3 있으면) |

### 핵심 차이 요약
- **문법**: Starlark는 "**파이썬처럼 생겼지만 부분집합**"(전역 부작용·while 제한·클래스 없음 등) — 이름변환 함수엔 충분하나 "진짜 Python 코드 재사용"은 안 됨. RustPython/CPython은 **진짜 Python**.
- **별도 설치**: **Starlark·RustPython 모두 불요**(우리 바이너리에 포함 → 사용자는 아무것도 안 깔아도 됨). **CPython(Tier B)만** 사용자 Python 필요.
- **실행 파일 크기**: Starlark **+1~3MB** ≪ RustPython **+8~15MB**. CPython(아웃오브프로세스)은 **우리 바이너리 증가 없음**(대신 사용자 런타임 의존).
- **성능**: 셋 다 이름변환(수천 파일 문자열 작업)엔 실사용 문제 없음. **경량·최속은 Starlark**, RustPython은 그보다 느리지만 CPython 배치와 대등, **CPython 인프로세스 아님**(격리 프로세스라 배치로 상각).

### 트레이드오프 결론
- **"진짜 Python 필수" (사용자 요구)** → **RustPython 기본**(+8~15MB 감수, 설치 0) + 필요 시 **CPython 옵션**(크기 0·설치 필요).
- **크기·속도가 최우선이고 파이썬 "유사"로 충분** → **Starlark**(+1~3MB·최속). 단 "진짜 Python 코드"는 아님.
- 즉 **Python 진위 ↔ 바이너리 크기**의 맞교환: 진짜 Python을 설치 없이 = **+8~15MB**(RustPython)가 그 대가.

## 9. 결정 기록(요약)
- **엔진**: 기본 **RustPython**(진짜 Python + 샌드박스 + 무런타임번들 + 결정성 + 맥 테스트), 옵션 **아웃오브프로세스 CPython**(패키지 필요 시). 착수 시 **ADR**로 확정(RustPython 성숙도·stdlib 커버리지 실측 포함).
- **보안 원칙**: 이름변환기는 **무 I/O 순수 함수**가 계약. Tier A는 강제, Tier B는 프로세스 격리 + "신뢰된 확장"으로 표기.
